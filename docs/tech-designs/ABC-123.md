# Technical Design: STR-16 - Add new Status: "Disputed" to the Orders

## Jira ticket

- Ticket: [STR-16 - Add new Status: "Disputed" to the Orders](https://softxpert.atlassian.net/browse/STR-16)
- Scope: Design only. Do not implement until a human explicitly approves this design.

## 1. Problem summary

Support users need dashboard visibility into orders whose Stripe payment transaction has become disputed. Today, an applicant can dispute a payment through their bank/Stripe and still continue the Sterling order flow, including fingerprint submission, because the application has no recorded disputed state for the order.

The feature should detect Stripe disputes for paid orders, mark affected orders as `Disputed`, preserve the prior order status, expose dispute metadata in order details, and reflect disputed orders across dashboard, statistics, and accounting views.

## 2. Current behavior

Based on the current repository contents available for review:

- The backend is a Spring Boot 3.5.x Maven application using Java 21.
- Existing dependencies include Spring Web, Spring Security, Spring Data JPA, Spring Validation, Flyway, MySQL, H2, Lombok, and Spring test libraries.
- The current visible persistence setup only includes an `app_users` Flyway migration.
- No existing order, payment, Stripe webhook, dashboard, statistics, accounting, or Sterling integration modules were found in the accessible code search results.

Functional current behavior from STR-16:

- Orders can remain actionable even after the associated Stripe payment is disputed or refunded externally.
- There is no reliable dashboard indicator showing that an order's payment has been disputed.
- Order status filters, statistics, accounting, and details views do not currently expose a `Disputed` status, `Disputed On`, or `Last Status` fields.

## 3. Proposed behavior

Add a Stripe dispute ingestion and order-state update flow.

### 3.1 Dispute detection

Implement a Stripe webhook endpoint that accepts dispute-related events and maps them back to the internal order through the Stripe payment identifier.

Recommended Stripe events to support:

- `charge.dispute.created` as the primary trigger to mark an order as disputed.
- `charge.dispute.closed` for follow-up handling, if product decides later whether won/lost disputes should keep or clear the `Disputed` state.

The first implementation should focus on marking orders disputed when a dispute is created. Do not auto-clear or auto-restore order status without explicit business approval.

### 3.2 Eligible orders

Only check or mutate orders when all of the following are true:

- The order is paid through Stripe, not on-account.
- The order type is one of the supported paid order families listed in STR-16: FBI, CA, FDLE, and other supported CA/Sterling order variants.
- The current order status is one of:
  - `New`
  - `Under Retrial`
  - `Completed`
  - `Resubmitted`

Do not check or mutate orders whose current status is:

- `Canceled`
- `Disputed`

### 3.3 State transition

When a Stripe dispute is associated with an eligible order:

1. Store the current order status as `lastStatusBeforeDispute`.
2. Set the current order status to `Disputed`.
3. Set `disputedOn` to the dispute event timestamp, falling back to server receipt time if Stripe event time is unavailable.
4. Store Stripe dispute metadata sufficient for idempotency and auditability.
5. Emit an audit/event log entry for support traceability.

### 3.4 Dashboard and order details

- Add `Disputed` to the order status filter dropdown.
- Show disputed orders in the dashboard grid with the `Disputed` status and grey color indicator.
- Add `Disputed On` and `Last Status` to order detail responses/views.
- Action availability on disputed orders should depend on `Last Status`:
  - Last status `New`: allow the existing Sterling cancel order action.
  - Last status `Under Retrial`: allow the existing cancel retrial policy action.
  - Last status `Completed` or `Resubmitted`: expose no action.

### 3.5 Reporting surfaces

- Include `Disputed` orders in the Statistics page aggregations.
- Include `Disputed` orders in the Accounting page, clearly separated from completed/paid revenue where applicable.

## 5. Data model or API changes

### 5.1 Order data model

Proposed order fields:

- `status`: add supported value `Disputed`.
- `lastStatusBeforeDispute`: nullable enum/string storing the order status immediately before the transition to `Disputed`.
- `disputedOn`: nullable timestamp with timezone.
- `stripeDisputeId`: nullable string, unique if present.
- `stripeDisputeStatus`: nullable string for Stripe's dispute lifecycle status.
- `stripeDisputeReason`: nullable string.
- `stripeDisputeAmount`: nullable integer/decimal, depending on existing money representation.
- `stripeDisputeCurrency`: nullable ISO currency code.
- `stripeDisputeEventId`: nullable string for idempotency/audit reference.

If a separate payment transaction table already exists, prefer storing Stripe-specific dispute metadata on the payment transaction and only storing order-level fields required for dashboard/query performance on the order.

### 5.2 Migration plan after approval

A future implementation PR should add Flyway migration(s) to:

- Add `Disputed` to the order status representation, if the database enforces status values.
- Add nullable dispute metadata columns to orders or payment transactions.
- Add an index on `status` for dashboard filtering if one does not already exist.
- Add an index/unique constraint on `stripe_dispute_id` or `stripe_dispute_event_id` for idempotency if stored in the database.

No migration is included in this design-only PR.

### 5.3 API additions/changes

Proposed backend API changes:

- `POST /api/webhooks/stripe`
  - Accepts Stripe webhook payloads.
  - Verifies Stripe signature using configured webhook secret.
  - Handles dispute events idempotently.
  - Returns `2xx` for known already-processed events to avoid webhook retries.
- Order dashboard/list endpoint:
  - Accepts or returns status value `Disputed`.
  - Supports filtering by `Disputed`.
  - Includes status color metadata or maps `Disputed` to the frontend's grey indicator convention.
- Order details endpoint:
  - Adds `disputedOn`.
  - Adds `lastStatusBeforeDispute`.
  - Adds action availability flags based on last status.
- Statistics endpoint:
  - Adds/handles `Disputed` in status counts.
- Accounting endpoint:
  - Includes disputed orders and classifies them separately from normal paid/completed totals.

## 6. Edge cases

- Duplicate Stripe webhook deliveries: handler must be idempotent by Stripe event ID and/or dispute ID.
- Out-of-order Stripe events: `charge.dispute.closed` may arrive before or after `charge.dispute.created`; initial implementation should not restore state automatically.
- Order already `Disputed`: do not overwrite `lastStatusBeforeDispute` or `disputedOn` unless explicitly designed for corrections.
- Order `Canceled`: do not change to `Disputed`; optionally log ignored dispute metadata for audit.
- On-account order: skip because no Stripe payment status applies.
- Missing payment/order mapping: return success to Stripe after logging/alerting, unless retrying can realistically resolve delayed consistency.
- Multiple payments for one order: define whether any disputed charge marks the entire order disputed; recommended default is yes for STR-16.
- Multiple disputes for one payment: preserve first disputed timestamp and store/update latest Stripe dispute status separately.
- Refund plus dispute: do not assume refund prevents dispute; rely on Stripe dispute event as source of truth.
- Last status action mapping: only expose actions for `New` and `Under Retrial`; no actions for `Completed`/`Resubmitted`.
- Time zones: store timestamps in UTC and localize only at presentation boundaries.
- Manual status changes after dispute: require explicit authorization and audit trail; avoid implicit transitions back to prior status.

## 7. Rollout/migration plan

1. Human reviews and approves this design.
2. Add the approved database migration(s) in a separate implementation PR.
3. Add backend enum/model/DTO/API changes.
4. Add Stripe webhook endpoint behind verified signature handling.
5. Configure Stripe webhook secret per environment.
6. Register the webhook in Stripe Dashboard for the selected events.
7. Deploy backend changes to a lower environment.
8. Use Stripe CLI/test mode to emit dispute events and verify dashboard, details, statistics, and accounting behavior.
9. Deploy to production with logging/metrics for dispute event handling.
10. Monitor ignored events, unmatched payments, duplicate deliveries, and disputed order counts.

Rollback strategy:

- Disable the Stripe webhook endpoint in Stripe Dashboard or remove the event subscription.
- Keep persisted dispute fields intact for auditability.
- If a logic defect incorrectly marks orders disputed, run a reviewed/manual correction using `lastStatusBeforeDispute` and audit logs.

## 8. Test plan

### Unit tests

- Dispute handler marks eligible orders as `Disputed`.
- Handler stores `lastStatusBeforeDispute` before changing status.
- Handler sets `disputedOn` from Stripe event time when available.
- Handler skips on-account orders.
- Handler skips `Canceled` and already `Disputed` orders.
- Handler is idempotent for duplicate Stripe event IDs.
- Action availability is correct for last statuses `New`, `Under Retrial`, `Completed`, and `Resubmitted`.

### Controller/API tests

- Stripe webhook rejects invalid signatures.
- Stripe webhook accepts valid dispute events.
- Dashboard status filter includes `Disputed`.
- Order list returns disputed status and grey indicator mapping.
- Order detail response includes `disputedOn` and `lastStatusBeforeDispute`.

### Repository/integration tests

- Orders can be queried by `Disputed` status.
- Statistics include disputed orders.
- Accounting includes disputed orders in the correct classification.
- Flyway migration applies cleanly on MySQL and H2/local profile after implementation.

### Manual/QA tests

- Register a test Stripe webhook endpoint.
- Trigger a test dispute event with Stripe CLI or test-mode Stripe Dashboard.
- Verify each STR-16 acceptance criterion against dashboard, details, statistics, and accounting pages.

## 9. Risks and open questions

### Risks

- The accessible repository currently does not show order/payment/dashboard modules, so file paths and exact integration points may need adjustment once those modules are available.
- Stripe event-to-order mapping may be non-trivial if payment identifiers are not stored consistently.
- Automatically marking completed/resubmitted orders as disputed may affect downstream Sterling workflows and reporting assumptions.
- Accounting totals may be misinterpreted unless disputed orders are clearly separated from normal revenue states.
- If webhook signature verification is skipped or misconfigured, the endpoint could allow unauthorized order-state changes.

### Open questions

- What exact Stripe object maps to an order today: `payment_intent`, `charge`, `checkout_session`, or another identifier?
- Are order statuses stored as database enum values, strings, lookup table rows, or application-only enum constants?
- Should `charge.dispute.closed` ever restore a previous status when the merchant wins the dispute?
- Should historical disputes be backfilled, or only future webhook events handled?
- What is the canonical list of order types, given STR-16 repeats `CA` and mentions FBI, CA, FDLE, CA?
- Should disputed completed/resubmitted orders block all downstream applicant/Sterling actions, or only hide support actions?
- Which page owns the grey status indicator mapping: backend DTO, frontend enum, or shared config?
